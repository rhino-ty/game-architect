# S-12: 절차적 생성 설계 (Procedural Generation)

> ⚠️ **에이전트/Claude Code 주의**
> 이 파일의 코드 블록은 **개념 설명용 의사코드(pseudocode)** 또는 **설계 참고용**이다.
> 게임에 직접 복사하지 말 것.
> 실제 구현은 게임의 장르·엔진·아키텍처에 맞게 새로 설계할 것.
> 밸런싱 수치 시뮬레이션 코드는 `ref/game-formulas.md` 참조.

## 절차적 생성의 3가지 거짓말

```
거짓말 1: "완전히 랜덤하면 재밌을 것이다"
  현실: 완전 랜덤은 불공정하고 지루함
  해결: 제어된 랜덤 (Controlled Randomness)

거짓말 2: "절차적 생성이 콘텐츠를 대체한다"
  현실: 절차적 생성은 콘텐츠를 배열하는 도구일 뿐
          좋은 재료가 없으면 조합해도 좋지 않음
  해결: 수작업 "재료" 품질이 알고리즘보다 먼저

거짓말 3: "시드만 같으면 항상 같은 경험"
  현실: 플레이어 선택에 따라 같은 시드도 다른 경험
  해결: 재현 가능성 vs 변동성 의도적으로 설계
```

## Part 1: 맵/레벨 생성 알고리즘 비교

알고리즘 선택은 게임 특성에 따라 다르다. 아래는 설계 판단용 비교.
**구현 코드는 엔진 스킬(unity.skill / godot.skill)에서 생성.**

### BSP (Binary Space Partitioning)
```
원리:
  공간을 이진 분할 → 각 분할 영역에 방 배치 → 방 간 복도 연결

적합한 상황:
  탑다운 던전 크롤러, Roguelite
  방이 명확히 분리된 구조 원할 때
  구현 난이도: 낮음 (가장 클래식)

결과 특징:
  규칙적이고 예측 가능한 구조
  "던전스러운" 느낌

설계 시 결정해야 할 것:
  최소 방 크기 / 최대 분할 깊이
  복도 연결 방식 (L형 vs 직선 vs 구불구불)
```

### Wave Function Collapse (WFC)
```
원리:
  타일 간 인접 규칙 정의 → 제약 전파로 전체 맵 생성

적합한 상황:
  자연스러운 환경 (숲, 도시, 동굴)
  타일셋이 복잡하고 다양한 조합이 필요할 때
  구현 난이도: 높음

결과 특징:
  유기적이고 자연스러운 배치
  손으로 만든 것처럼 보임

주의:
  제약 충돌 시 생성 실패 가능 → fallback 로직 필수
  큰 맵에서 성능 이슈 가능
```

### Noise 기반 지형 (Perlin/Simplex Noise)
```
원리:
  좌표(x,y)를 noise 함수에 넣어 연속적인 높이값 생성 → 지형 분류

적합한 상황:
  오픈월드 자연 지형 / 섬, 산맥, 강
  구현 난이도: 낮음 (라이브러리 사용)

설계 시 결정해야 할 것:
  scale:    낮을수록 지형이 넓고 부드러움
  octaves:  높을수록 디테일 증가 (성능 비용)
  높이 임계값: 물/모래/풀/산/눈의 경계값 (게임에 맞게)
```

### Room Prefab 조합 (하데스 방식)
```
원리:
  수작업으로 만든 방 조각 풀에서 랜덤 선택 후 조합

적합한 상황:
  방 품질이 중요한 Roguelite
  "좋은 랜덤"이 필요할 때
  구현 난이도: 낮음 (알고리즘보다 콘텐츠 작업)

결과 특징:
  항상 플레이 가능한 레이아웃 보장
  디자이너 의도 반영 가능

하데스가 이 방식을 선택한 이유:
  "완전 절차적 생성보다 수작업 방이 더 재밌다"
  → 1인 개발에도 강력히 권장
```

## Part 2: 아이템/보상 생성 설계 개념

### 가중치 기반 선택 (Weighted Random)

```
개념:
  단순 확률(50/50)이 아닌, 각 아이템에 "출현 빈도 가중치" 부여.

설계 예시:
  일반 아이템:  가중치 60
  고급 아이템:  가중치 25
  희귀 아이템:  가중치 10
  전설 아이템:  가중치  5

  ※ 비율이 아닌 상대적 무게. 합계 100 불필요.

설계 시 결정:
  전체 아이템 풀 크기 (너무 작으면 반복, 너무 크면 희석)
  상황별 다른 풀 (보스방 / 상점 / 보물상자 각각)
  Luck 스탯 연동 여부

[의사코드 — 개념 참고용]
pool = [(아이템A, 60), (아이템B, 25), (아이템C, 10), (아이템D, 5)]
total = 각 가중치의 합
roll = random(0, total)
cumulative = 0
for each (item, weight) in pool:
    cumulative += weight
    if roll <= cumulative: return item
```

### 희귀도 계층 시스템

```
표준 5계층:
  일반(Common):    ~60%  — 항상 쓸 수 있는 기반
  고급(Uncommon):  ~25%  — 일반보다 확실히 좋음
  희귀(Rare):      ~10%  — 빌드를 바꿀 수 있음
  영웅(Epic):       ~4%  — 강력한 특수 효과
  전설(Legendary):  ~1%  — 빌드 정체성을 결정

  ※ 위 %는 참고용. 게임 경제에 맞게 시뮬레이션 후 조정.
  → 시뮬레이션 코드: skills/numeric-simulation.md

전설 아이템 설계 원칙:
  수치가 아니라 메커닉이 달라야 함
  "공격력 +500%"가 아니라 "공격이 모든 방향으로 발사됨"

Pity(천장) 시스템:
  N회 이상 전설 미획득 시 보장
  → 구체적 N값은 기댓값 계산 후 결정
  → skills/numeric-simulation.md의 가챠 시뮬레이션 참조
```

### 선택지 품질 보장

```
나쁜 선택지 상황:
  "이미 가진 것들만 나온다" → 유저 좌절

보장 설계 원칙:
  3개 선택지 중 최소 1개는 현재 빌드에 없는 것
  3개 선택지 중 최소 1개는 현재 빌드와 시너지 있는 것

[의사코드 — 개념 참고용]
choices = []
choices.add( 현재_보유_목록에_없는것_랜덤() )     // 새로운 방향 보장
choices.add( 현재_빌드와_시너지_있는것_랜덤() )   // 기존 빌드 강화
choices.add( 전체_풀에서_완전_랜덤() )             // 예상 외 선택
shuffle(choices)
```

## Part 3: 시드 기반 재현 가능성 설계

```
왜 필요한가:
  버그 재현: "이 시드로 하면 크래시 납니다" 가능
  공유: "이 런 진짜 좋았어" 시드 공유
  데일리 챌린지: 모든 유저 같은 시드

설계 결정 사항:
  시드 = 날짜 + 런 번호 + 플레이어 ID 조합
  → 같은 날 같은 사람이 하면 항상 같은 런

무엇을 시드로 제어하는가:
  방 배치, 아이템 풀, 적 배치 → 시드로 고정
  플레이어 선택 (어떤 아이템 집는지) → 시드 무관

구현 원칙:
  모든 랜덤 호출은 동일한 시드 기반 RNG 객체 사용
  전역 random() 함수 직접 호출 금지 → 시드 통제 불가능

  [의사코드]
  rng = new SeededRandom(run_seed)
  room_layout = rng.generateRooms()    // 시드 종속
  item_pool   = rng.pickItems()        // 시드 종속
  player_choice = getPlayerInput()     // 시드 무관
```

## Part 4: Roguelite 런 설계

### 난이도 곡선

```
목표: 런의 처음부터 끝까지 긴장감 유지

권장 구조 (하데스 참고):
  구역 1~3:  쉬움 — 핵심 메커닉 재확인, 빌드 방향 탐색
  구역 4~6:  중간 — 빌드 시너지 형성
  구역 7~9:  어려움 — 빌드 완성, 실수 허용 범위 좁아짐
  보스:      고난이도 — 빌드 활용 테스트

조정 변수 (구역마다 조정):
  적 체력: +10~20%
  신규 적 패턴: 최소 1개
  엘리트 출현 비율: 초반 5% → 후반 20%

  ※ 위 수치는 출발점. 실제 플레이테스트로 조정.
```

### 선택지 빈도

```
너무 드물면: 빌드 다양성 감소
너무 잦으면: 결정 피로

참고 기준:
  방 3~5개마다 선택 기회 1회
  보스 직전: 반드시 선택 기회
  상점: 골드로 직접 구매 (랜덤 대비 안전판)
```

## Part 5: 생성 실패 대응 (필수)

```
절차적 생성은 항상 실패할 수 있다.
실패 처리 없는 구현 = 미완성.

대응 전략:

1. 재시도 한계 설정
   최대 시도 횟수(예: 10회) 초과 시 fallback 실행
   무한 루프 = 게임 멈춤

2. Fallback 설계
   A: 수작업 기본 레이아웃으로 대체
   B: 더 단순한 알고리즘으로 전환
   C: 인접 시드로 재시도

3. 연결성 검증
   생성 후 반드시 시작점 → 목표점 경로 존재 확인
   (BFS 또는 Flood Fill로 검증)

4. 비동기 생성
   큰 맵 실시간 생성 → 프레임 드롭 위험
   → 로딩 화면 중 사전 생성
   → 청크 스트리밍 (플레이어 주변만 생성)
```

## 알고리즘 선택 결정 가이드

```
내 게임에 맞는 알고리즘:

탑다운 던전 + 방이 명확:     BSP 또는 Room Prefab
자연스러운 야외 지형:         Noise 기반
아트 품질 + 플레이 보장:      Room Prefab (하데스 방식)
복잡한 타일 환경 규칙:        WFC
1인 개발 + 빠른 구현:         Room Prefab 강력 권장
```
