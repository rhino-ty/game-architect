# S-11: 출시 후 운영 (Post-Launch Operations)

## 목적
출시는 끝이 아니라 시작이다.
출시 후 첫 2주, 첫 3개월, 첫 1년의 운영 방식이
게임의 총 매출과 커뮤니티 신뢰도를 결정한다.

---

## 출시 직후 (Day 0~3): 긴급 대응 체계

### 모니터링 대시보드
```
매 2시간 확인:
  □ Steam 리뷰 새 글 (긍정/부정 비율)
  □ Steam 토론 탭 (버그 신고, 불만)
  □ Discord #bug-reports
  □ Twitter/Reddit 멘션 (@게임명)

Steam 리뷰 실시간 확인 도구:
  steam 페이지 → 리뷰 탭 → 최신순 정렬
  SteamDB.info 에서 리뷰 추이 그래프 확인
```

### 버그 대응 우선순위
```
P0 — 즉각 핫픽스 (1~6시간 내):
  크래시 / 진행 불가 버그 / 데이터 손실
  → 패치 없이 부정 리뷰 폭탄 시작됨

P1 — 당일 패치 (24시간 내):
  주요 기능 오작동 / 심각한 밸런스 이탈
  → Steam 공지 필수

P2 — 빠른 패치 (3~7일 내):
  소규모 버그 / 불편사항
  → 패치 묶어서 한번에

P3 — 다음 업데이트에 포함:
  개선 요청 / 미세 조정
```

### 부정 리뷰 대응
```
Steam은 개발자가 리뷰에 공개 답변 가능.

답변 원칙:
  □ 24~48시간 내 답변
  □ 방어적 태도 절대 금지
  □ 구체적인 해결 계획 또는 인정

좋은 답변 예시:
  "이 문제를 직접 겪으셨군요. 정확히 어떤 상황인지 파악했습니다.
  [날짜] 패치에서 수정 예정입니다. 불편을 드려 죄송합니다."

피해야 할 답변:
  "다들 잘 되는데요?" (방어적)
  "그건 버그가 아닙니다" (부정)
  "곧 고치겠습니다" (모호한 약속)
```

## 출시 후 첫 달: 안정화 & 기반 구축

### 첫 번째 콘텐츠 업데이트 (출시 후 2~4주)
```
목적: "개발자가 게임을 버리지 않았다"는 신호
내용: 버그 수정 묶음 + 소규모 개선 + 커뮤니티 요청 1~2개 반영

공지 형식:
  제목: "패치 1.0.1 — 커뮤니티 피드백 반영"
  내용:
    - 수정된 버그 목록
    - 개선된 사항
    - "다음에 작업할 것" 예고
  
  채널: Steam 공지 + Discord + Twitter
```

### Steam 리뷰 상태 관리
```
리뷰 상태 분류:
  압도적 긍정적 (95%+): 최상, 유지
  매우 긍정적 (80~94%): 양호
  대체로 긍정적 (70~79%): 주의 필요, 원인 파악
  복합적 (40~69%): 심각, 즉각 대응 필요
  부정적 (40% 미만): 위기

부정 비율 상승 시 체크:
  □ 특정 버그/이슈가 집중되는가? → 패치
  □ 가격에 대한 불만인가? → 세일 or 가격 조정
  □ 기대와 다른 게임이라는 불만? → 스토어 설명 수정
  □ 특정 플랫폼(GPU/OS) 문제? → 기술 지원 집중
```

## 얼리 액세스 → 1.0 전환 전략

### 전환 시점 결정 기준
```
체크리스트 (모두 충족 시 1.0 출시):
  □ 계획된 모든 핵심 컨텐츠 구현 완료
  □ 크리티컬 버그 0개
  □ 주요 버그 5개 이하
  □ 최소 30명 외부 플레이테스터 완료
  □ 평균 플레이타임이 기대 플레이타임의 70%+
  □ 현지화 완료 (최소 계획 언어)
  □ 성능 최적화 완료 (목표 FPS 안정적)

1.0이 아니어도 되는 이유:
  → 얼리 액세스가 길수록 커뮤니티가 함께 게임을 만든다는 느낌
  → 하데스: 2년 EA → 1.0에서 압도적 긍정적
  → 단, EA가 너무 길면 "개발 중단?" 인식 위험
```

### 1.0 출시 마케팅 (재출시 효과)
```
Steam 알고리즘:
  얼리 액세스 → 정식 출시 = 새 게임처럼 알고리즘 재부스트
  → 위시리스트 알림 재발송
  → 출시 큐레이션 재노출

마케팅 전략:
  □ 1.0 전용 트레일러 제작 (EA 트레일러와 다르게)
  □ "이제 완성됐다" 메시지 — EA 관망하던 유저 유입
  □ 미디어/스트리머 재피칭 (1.0 기준으로)
  □ 첫 번째 대형 세일 (1.0 출시와 동시 or 직후)
```

## Steam 세일 전략

### 세일 타이밍 최적화
```
Steam 주요 세일 시즌:
  여름 세일:    6월 말 ~ 7월 초 (연중 최대)
  가을 세일:    11월 말 (블랙프라이데이)
  겨울 세일:    12월 말 ~ 1월 초 (연중 2위)
  Spring 세일:  3~4월

첫 세일 타이밍:
  출시 후 최소 3개월 후 첫 세일 권장
  (너무 빠른 세일 = 정가 구매자 배신감)

할인율 가이드:
  첫 번째 세일:  20~30%
  이후 세일:     33~50%
  주년 기념:     50~66%
  번들 포함:     75%+ 가능

세일 중 할 일:
  □ 소셜 미디어 세일 공지
  □ Steam 공지 게시
  □ 커뮤니티 이벤트 연계 (세일 기간 이벤트)
```

### 번들 전략
```
Humble Bundle / Fanatical 번들:
  장점: 대규모 노출, 새 유저 유입
  단점: 게임당 수익 매우 낮음 ($0.1~$0.5)
  목적: 수익보다 커뮤니티 확장 & 리뷰 수 증가

Steam 번들 (자체):
  자신의 DLC + 메인 게임 번들
  또는 유사 인디 게임 끼리 크로스 번들
```

## 장기 운영 전략 (1년+)

### 업데이트 로드맵 공개
```
공개 로드맵의 장점:
  - 커뮤니티 기대감 유지
  - "게임 살아있다" 신호
  - 위시리스트 유지율 상승

공개 형식:
  Steam 공지 + 이미지로 로드맵 시각화
  분기별 업데이트 → 세부 내용은 나중에

약속 시 주의:
  □ 구체적 날짜보다 분기 단위로 (지연 리스크)
  □ 취소 가능성 있는 기능은 "검토 중" 표현
  □ 실제로 하지 않을 것은 절대 약속 금지
```

### 무료 업데이트 vs 유료 DLC 결정 기준
```
무료 업데이트:
  버그 수정, 밸런스, QoL 개선 → 항상 무료
  소규모 컨텐츠 추가 (맵 1개, 무기 1개) → 무료 권장
  출시 첫 6개월 컨텐츠 → 무료가 커뮤니티 신뢰 형성

유료 DLC:
  독립적인 스토리 챕터
  대규모 신규 구역 (메인 게임의 30%+ 분량)
  새로운 플레이어블 캐릭터 + 고유 메커닉

가격 기준:
  소규모 DLC: 메인 게임의 20~30%
  대규모 DLC: 메인 게임의 40~60%
  완전한 확장팩: 메인 게임의 50~70%
```

### 장기 커뮤니티 건강 지표
```
매월 확인:
  Steam 리뷰 평점 추이 → 하락 원인 분석
  Steam 동시 접속자 수 (SteamDB) → 리텐션 확인
  Discord 활성 멤버 수 → 커뮤니티 활력
  Reddit 언급 → 자연발생 입소문

경고 신호:
  동시 접속자 3개월 연속 감소 → 콘텐츠 업데이트 필요
  리뷰 부정 비율 증가 → 원인 파악 즉각 대응
  Discord 메시지 감소 → 이벤트 or 개발자 참여 필요
```

## 속편 / 스핀오프 판단 기준

```
속편 개발 신호:
  □ 메인 게임 리뷰 수 1,000개+ (커뮤니티 기반 충분)
  □ 매출이 다음 프로젝트 개발비 커버 가능
  □ 유저가 직접 속편을 요청하는 댓글/리뷰 다수
  □ 게임 세계관/메커닉에 아직 탐구 안 된 공간 있음

스핀오프 고려 시:
  □ 메인 메커닉 중 특히 반응 좋은 것을 분리
  □ 더 좁고 깊은 경험으로 재설계

신규 IP 전환 시:
  □ 기존 팬 베이스를 새 게임으로 자연스럽게 이동
  □ 메일링 리스트 / Discord 연락망 미리 구축
```

## 수익 분석 & 세금/정산

```
Steam 수익 정산:
  Steam 수수료: 30% (매출 $10M 이하)
  $10M~$50M: 25%, $50M+: 20%
  정산 주기: 월 1회 (월말 기준 D+30)

한국 세금 (개인 개발자):
  Steam 수익 = 외화 수입 → 종합소득세 신고 대상
  VAT(부가세): 소규모 면세 여부 확인 (연 매출 8천만원 미만 시 간이과세)
  → 출시 전 세무사 1회 상담 강력 권장

추적 도구:
  SteamSpy: 판매량 추정 (공개 데이터)
  SteamDB: 동시 접속자, 리뷰 추이
  Steam 판매 대시보드: 실제 수익 (개발자만 접근)
```
