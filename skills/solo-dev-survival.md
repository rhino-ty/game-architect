# S-13: 솔로 개발 생존 전략 (Solo Dev Survival)

## 목적
1인 개발에서 게임을 완성하지 못하는 가장 큰 이유는 기술 부족이 아니다.
스코프 크리프, 재시작의 루프, 번아웃 — 이 세 가지가 대부분의 인디 게임을 죽인다.

> "인디 게임 개발의 첫 번째 규칙: 게임을 완성하는 것이 좋은 게임을 만드는 것보다 먼저다."
> — Derek Yu (Spelunky 제작자, "Finishing a Game" 블로그)

---

## Part 1: 스코프 관리 (Scope Management)

### 스코프 크리프의 작동 방식

```
초기 아이디어: "간단한 Roguelite"
  ↓ 2주 후
"전투 시스템에 스태거 추가하면 어떨까?"
  ↓ 1개월 후
"NPC 대화 시스템도 넣어야 하지 않나?"
  ↓ 3개월 후
"오픈월드로 바꾸면 더 재밌을 것 같은데..."
  ↓ 무한 반복
개발 중단
```

이것이 Tom Cargill의 90/90 법칙으로 이어짐:
*"게임의 첫 90%가 개발 시간의 90%를 먹는다. 나머지 10%가 또 90%를 먹는다."*

### 스코프 제어 도구

**Feature Freeze Line (기능 동결선)**
```
명시적으로 날짜를 정한다:
  "YYYY년 MM월 DD일 이후 신규 기능 추가 없음"

동결 후 허용되는 것:
  ✓ 버그 수정
  ✓ 기존 기능 폴리시
  ✓ 밸런스 조정

동결 후 금지되는 것:
  ✗ 새로운 시스템
  ✗ 새로운 캐릭터/레벨
  ✗ "이거 하나만 더" 유혹
```

**MVP 정의를 처음에 못 박기**
```
시작 전에 "이것만 있으면 출시할 수 있다"를 문서화:

MVP 체크리스트 예시:
  □ 플레이어가 이동하고 공격할 수 있다
  □ 적이 3종류 이상 있고 각각 다른 패턴이다
  □ 보스가 1마리 있다
  □ 레벨이 5개 이상 있다
  □ 시작과 끝이 있다
  □ 저장/불러오기가 된다

이 리스트에 없는 것은 "다음 게임" 또는 "DLC"
```

**기능 요청 처리 프로세스**
```
새 아이디어가 떠오르면:

1단계: 메모장에 적는다 (버리지 않음)
2단계: 48시간 기다린다
3단계: MVP에 없다면 → "다음 게임 아이디어" 파일로 이동
4단계: MVP에 필수적이라면 → 대신 제거할 기능을 결정

"추가"가 아니라 "교환". 총량은 변하지 않는다.
```

## Part 2: 재시작의 루프 (Loop of Restarting)

Derek Yu가 명명한 1인 개발의 가장 치명적인 패턴.

### 왜 발생하는가
```
개발 6개월 차:
  실력이 올라감 → 초반 코드/그래픽이 마음에 안 들기 시작
  "이 정도 퀄리티론 출시 못해" → 처음부터 다시
  
재시작 후 6개월:
  또 실력이 올라감 → 또 초반이 마음에 안 들기 시작
  → 무한 반복
  
결과: 3년이 지나도 게임 없음
```

### 탈출 방법

**"프로토타입과 제품 분리" 원칙**
```
프로토타입:
  실험용. 코드 품질 신경 쓰지 않음.
  목적: "이 아이디어가 재밌는가?" 검증
  기간: 1~2주

프로덕션:
  이제 재밌다고 확인된 것만.
  처음부터 제대로 짠다.
  
프로토타입을 프로덕션으로 업그레이드하려는 유혹 → 대부분 재작성이 빠름
```

**"완성이 완벽보다 먼저" 원칙**
```
지금 코드가 나빠도 괜찮다. 나중에 고친다.
지금 그래픽이 나빠도 괜찮다. 나중에 바꾼다.
지금 없는 기능이 있어도 괜찮다. 나중에 추가한다.

완성하지 못하면 아무것도 배울 수 없다.
플레이어 반응 없이는 무엇이 잘못됐는지 모른다.

→ 어떤 게임이든 완성해서 출시한 경험 자체가 다음 게임의 가장 큰 자산
```

**마일스톤 기반 재설계 시점 제한**
```
리팩토링(코드 구조 개선)은 마일스톤 사이에만:
  알파 완성 → 리팩토링 허용
  베타 완성 → 리팩토링 허용
  개발 중간에 갑자기 → 금지

"지금 당장 고쳐야 해" 느낌 → 메모하고 다음 마일스톤까지 기다리기
```

## Part 3: 번아웃 예방

### 번아웃의 전조 증상
```
- 게임 켜는 게 두려워짐
- 작은 버그 하나도 고치기 싫음
- "차라리 다른 게임 하고 싶다"
- 프로젝트 생각만 해도 스트레스
- 자신의 게임이 나쁘다는 확신이 강해짐
  (대부분 사실이 아님 — 번아웃의 인지 왜곡)
```

### 예방 전략

**작업 시간 상한선**
```
일반 직장인 설정:
  하루 4~6시간 게임 개발
  하루 2~4시간 휴식/다른 취미

"오늘 8시간 했으니 내일도 8시간" → 2주 후 번아웃

지속 가능한 패턴:
  주 5일 × 하루 4시간 = 20시간
  이게 주 7일 × 6시간(+번아웃)보다 장기적으로 더 많은 작업량
```

**소규모 달성의 기록**
```
"오늘 한 일" 일지:
  × "전투 시스템 완성" (너무 큰 목표)
  ○ "적 피격 애니메이션 완성" (하루 완료 가능)
  
매일 완료 가능한 크기로 목표 분해.
완료한 것들을 지우지 않고 기록.
→ "나는 진전하고 있다" 감각 유지
```

**의도적인 휴가**
```
6~8주 개발 후 1주 완전 휴식 (의무)
게임 생각 금지, 다른 게임 플레이, 여행, 운동

역설: 쉬고 나서 돌아오면 버그가 보이고, 아이디어가 생기고, 속도가 빨라짐
쉬지 않으면: 같은 자리를 맴돌다 결국 중단
```

**커뮤니티 연결 유지**
```
혼자 개발하면 피드백 없이 자기 의심에 빠짐

최소한의 연결:
  Discord 인디 개발자 서버 1~2개
  Twitter/X #indiedev 주 1회 공유
  
효과:
  - "나만 이렇게 힘든 게 아니구나"
  - 외부 피드백으로 인지 왜곡 교정
  - 데드라인 감각 (공개된 진행 상황이 동기 부여)
```

## Part 4: 1인 개발 현실적 타임라인

### 프로젝트 규모별 현실적 기간

```
마이크로 프로젝트 (Jam 게임 규모):
  플레이타임: 30분~2시간
  개발 기간:  1~4주
  목적:       완성 경험, 기술 학습
  출시:       itch.io 무료 배포

소규모 인디 (첫 상업 게임 권장):
  플레이타임: 3~6시간
  개발 기간:  6~12개월
  가격:       $4.99~$9.99
  출시:       Steam

중규모 인디:
  플레이타임: 8~20시간
  개발 기간:  12~24개월
  가격:       $9.99~$19.99
  출시:       Steam + 콘솔 고려

대규모 인디 (솔로는 권장하지 않음):
  플레이타임: 20시간+
  개발 기간:  24~48개월
  팀 없이는 번아웃 확률 매우 높음
```

### "첫 게임을 크게 만들지 마라"는 이유

```
스타듀밸리 착시 현상:
  "혼자 4년 만든 스타듀밸리가 대성공!"
  → 착시. Eric Barone은 그 전에 이미 게임 개발 경험이 있었음
  → 스타듀밸리가 첫 게임이었다면 미완성으로 끝났을 가능성 높음

현실:
  첫 게임의 목표 = 완성해서 출시하는 것
  두 번째 게임 = 더 크게 만들 수 있음
  세 번째 게임 = 이제 스타듀밸리급을 생각해도 됨

완성된 소규모 게임 1개 > 미완성 대규모 프로젝트 10개
```

## Part 5: 작업 시스템 (1인 개발용)

### 주간 루틴 예시

```
월요일: 이번 주 목표 3가지 결정 (메모에 적기)
화~목:  집중 개발 (하루 4~6시간)
금요일: 이번 주 완료된 것 기록, 플레이테스트
토요일: 마케팅/커뮤니티 (트위터 공유, 데브로그)
일요일: 완전 휴식

월요일에 목표 3개 이상 잡지 않기 (항상 과대평가)
```

### 개발 세션 구조

```
세션 시작 (5분):
  오늘 할 것 1가지 명시 ("피격 SFX 5종류 제작")
  
집중 시간 (90분):
  그것만 한다. SNS, 유튜브 금지.
  Pomodoro 25분 × 3세트 또는 자신에게 맞는 방식

세션 종료 (5분):
  완료 여부와 상관없이 기록
  내일 할 것 1가지 미리 정하기

  → 세션을 "어디서 멈췄는지"를 기록해두면
    다음 세션 진입 시간이 줄어듦 (워밍업 감소)
```

### 막힐 때 대처법

```
코드 막힘:
  1. 30분 더 시도
  2. 안 되면 → 다른 기능 먼저 (막힌 것 옆에 두기)
  3. 다음 날 신선한 눈으로 다시
  4. Stack Overflow, Discord, Reddit에 질문

동기 막힘 (하기 싫음):
  1. 가장 쉬운 것부터 시작 (1분이면 끝나는 것)
  2. 시작하면 대부분 계속하게 됨
  3. 그래도 안 되면 → 오늘은 쉬는 날

방향 막힘 (뭘 해야 할지 모름):
  1. GDD(게임 기획서) 열기 → 현재 단계 확인
  2. 다른 인디 개발자에게 피드백 요청
  3. 24시간 이상 지속되면 → 커뮤니티에 공유하고 외부 시각 구하기
```

## Steam 2024-2025 냉정한 현실

> 검색 데이터 기반 — 출처: GameDiscoverCo, VGI 2024-2025 연구

```
2024년 출시된 인디 게임 중 재정적으로 의미 있는 수준에 도달한 비율:
  약 0.5% 미만

위시리스트 → 첫 달 판매 전환율 중앙값:
  약 27% (얼리 액세스는 약 20%)

위시리스트 10,000개 이상 게임 첫 주 전환율:
  중앙값 약 0.15x → 위시리스트 1만 개면 첫 주 1,500판매

Steam 페이지 오픈 후 출시까지 기간:
  성공 게임 평균: 214일
  저조 게임 평균: 411일 (너무 오래 기다리면 관심 식음)

이것이 의미하는 것:
  위시리스트가 많아도 보장이 없다
  위시리스트가 적어도 바이럴이면 대박이 난다 (Nubby's 사례)
  결국 게임의 재미 + 마케팅의 타이밍이 둘 다 필요
  
  → 성공 확률에 집착보다 "완성하고 출시"에 집중
  → 첫 게임의 목표는 학습과 경험, 두 번째 게임부터 기대
```
